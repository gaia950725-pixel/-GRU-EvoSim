## [4] 개발 철학 및 코딩 규약

### 최우선 가치 5원칙 (구체화)
1. **성능**: 다수 에이전트·장기 플레이에서 병목을 만들지 않는다.
2. **유지보수 용이성**: 가독성 향상을 위해 함수 위치 재정렬/응집도 개선을 허용한다.
3. **내러티브 정합성**: 인류학/생태학 발전사와 충돌하는 단순화는 경고하고 근거를 남긴다.
4. **단순성**: 밸런싱/튜닝이 쉬운 함수 설계를 선호한다.
5. **확장 가능성**: 1회성 코드보다 후속 시스템 확장을 고려한 인터페이스를 우선한다.

### 가치 충돌 처리 규칙
- 충돌 발견 시 반드시 정책요청을 올린다.
- 정책요청에는 대안 2개 이상 + 우선순위(1~5) 반영한 **권고안 1개**를 포함한다.

### Fail-fast 및 편집 규약
- Fail-fast 원칙 유지.
- BUILDER_BLOCK 단위 편집 원칙 유지.

## [5] 프로세스 및 산출물 규격
- 정석 프로세스: 아이디어회의 → 작업로드맵 → 코드현황조사 → 청사진 → 기획서 → 정책확정(사용자) → 패치로드맵 → 작업명세 → 프롬프트 → 실행·실행보고 → 코드점검 → 사용자점검
- 간이 프로세스: 아이디어회의 → 프롬프트 → 패치실행 → 사용자점검 (사용자 요청 시 단계 추가)
- 산출물 상세 필드는 `docs/spec/SPEC_OUTPUTS.md` 참조.
