# EvoSim v27.28 Phase 1 — Config Screen Code Audit (Revised)

Base file: `EvoSim_27.28_builder-block_GAP-finalize_FINAL.html`
Audience: planning team + development team
Status: revised after policy confirmation

---

## 0. Scope of this revision

This revised Phase 1 reflects three fixed policies.

1. **Phase 1 / first implementation keeps current `id/name` based collection.**
   - No `data-cfg-*` migration in the first patch.
   - `data-cfg-*` may be introduced only in a later cleanup patch.

2. **Former player-facing controls that were later deprecated must not remain in the config screen after the patch.**
   - Targets:
     - `cfg-pred-bonus`
     - `predThreat`
     - `cfg-herb-bonus`
     - `cfg-herb-threat`

3. **Removing those controls does not mean removing the values themselves.**
   - Their values must survive as engine-owned fixed defaults, like other internal constants.
   - They are removed from player config DOM and from config collection, but not deleted from engine behavior.

This document therefore distinguishes between:
- what exists **now** in v27.28,
- what remains **player-facing** after the patch,
- what must be re-owned as **engine/internal fixed defaults**.

---

## 1. Executive findings

### 1-1. `guardedStart()` is still the main coupling point
`guardedStart()` directly reads config DOM, normalizes values, writes some UI back, persists some preferences, and prepares worldgen inputs.

That means config card edits still risk breaking start logic.

### 1-2. Config preview logic is still fragmented
The preview/UI dependency logic is currently split across:
- `_syncClimateUI()`
- `_syncAltUI()`
- `_syncMarUI()`
- `_syncMapDependentUI()`
- `_uiTouchedRiver`
- `_uiTouchedMtn`

So config UI is not just display; it already contains decision logic.

### 1-3. Some hidden controls are still live configuration inputs
The following are hidden/disabled for the player but are still read by runtime logic:
- `cfg-pred-bonus`
- `predThreat`
- `cfg-herb-bonus`
- `cfg-herb-threat`

This is now policy-incompatible. They must stop being config-screen inputs.

### 1-4. First patch should not widen the migration surface
Because the current start path is still tightly coupled to DOM ids/names, first implementation should keep `id/name` collection and only move collection behind a centralized pipeline.

## 2. Ownership model for the revised plan

| Ownership | Meaning | Player-facing after patch? |
|---|---|---|
| `SimulationConfig` | Start-time settings chosen by the player and applied to `CFG` | Yes |
| `DerivedPreviewState` | Preview/UI-only derived state inside the config screen | Yes, as output only |
| `ConfigUIState` | Config screen interaction state such as active tab and touched flags | Yes, internal to config UI |
| `EngineInternalFixedDefaults` | Engine/internal fixed values no longer exposed as player controls | No |
| `NotAppliedYet` | Present in UI or markup but not actually part of the start pipeline | No, cleanup candidate |

Important distinction:
- `EngineInternalFixedDefaults` are **not** hidden player settings.
- They remain part of engine behavior.
- They must not remain under player config DOM after the patch.

## 4. Controls that must be removed from player config UI

> These controls exist now in v27.28, but must not remain in the config screen after the patch.

| Current DOM id / name | Current runtime target | Current status in v27.28 | Required post-patch disposition |
|---|---|---|---|
| `cfg-pred-bonus` | `CFG.predatorKillBonus` | hidden/devonly, disabled, still read at start | remove from config DOM; do not collect from UI; source from engine fixed default |
| `name="predThreat"` | `CFG.pred2_threatLevel` | hidden/devonly radio, disabled, restored from localStorage, still read at start | remove from config DOM; stop player restore/persist; source from engine fixed default |
| `cfg-herb-bonus` | `CFG.herbivoreBonus` | hidden/devonly, disabled, still read at start | remove from config DOM; do not collect from UI; source from engine fixed default |
| `cfg-herb-threat` | `CFG.herbivoreThreatLevel` | hidden/devonly, disabled, still read at start | remove from config DOM; do not collect from UI; source from engine fixed default |

### Policy consequence
These are **not** “temporarily hidden player settings.”
They are **deprecated player controls** that now belong to engine-owned fixed defaults only.

## 6. Current runtime order that matters for refactor planning

### A. Immediate preview path (before start)
- climate checkbox updates preview state and disables/enables related controls
- altitude checkbox updates preview state and mountain control state
- maritime checkbox updates preview state text only
- map type recomputes river/mountain max/default and farm % preview
- sliders update labels immediately
- active tab is remembered in localStorage

### B. Start path (`guardedStart()`)
Current start path still mixes these steps:
1. read player config DOM
2. read hidden deprecated internal controls from DOM
3. normalize/fallback/clamp
4. write some normalized values back into DOM
5. write some preferences to localStorage
6. compute worldgen inputs from config
7. continue reset/worldgen/start

This mixed sequence is the exact coupling point the next phases must remove.

---

## 7. First-patch constraints derived from this audit

1. **Keep `id/name` based collection in first implementation.**
   - centralize DOM reads, but do not migrate markup to `data-cfg-*` yet.

2. **Remove deprecated internal controls from the config screen DOM.**
   - no hidden placeholders left in player config markup.
   - no start-time reads from those DOM nodes.

3. **Replace deprecated control values with engine-owned fixed defaults.**
   - `CFG.predatorKillBonus`
   - `CFG.pred2_threatLevel`
   - `CFG.herbivoreBonus`
   - `CFG.herbivoreThreatLevel`
   must continue to exist, but not as player-adjustable config.

4. **Stop player-facing restore/persist for deprecated internal controls.**
   - especially `pred2_threatLevel` localStorage restore/persist.

5. **Keep existing player-visible semantics stable.**
   - species fallback
   - mapType-based river/mountain limits/defaults
   - altitude off forcing mountain count to zero
   - current leader/clan selection semantics

6. **Do not touch in-game panel layout/state in this refactor.**
   - `.panel`
   - `info/history/inspector`
   - `__HS_UI`
